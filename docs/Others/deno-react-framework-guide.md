## 一、三種開發方式總覽

用 Deno 開發跨平台應用，依照目標平台不同，主要有三種方向：

### 1. Deno + Vite + React（網頁應用）

- **定位**：跑在瀏覽器的 SPA，開發體驗最接近傳統前端
- **跨平台方式**：任何有瀏覽器的裝置都能用
- **部署**：靜態檔案，可放到任何主機
- **存取系統資源**：無法（無法讀寫本機檔案、系統 API）
- **適合**：Web 應用、SPA、後台系統

### 2. Deno Fresh 框架（全端應用）

- **定位**：SSR（伺服器端渲染）網頁應用，Deno 官方推薦框架
- **跨平台方式**：透過瀏覽器存取，Server 跑在任何平台
- **部署**：需要 Deno 伺服器環境，可部署到 Deno Deploy
- **存取系統資源**：Server 端可以（資料庫、檔案系統等）
- **特色**：零 JS 預設、Islands 架構、SEO 友善
- **適合**：需要 SEO、有後端邏輯的全端應用

### 3. Deno + Vite + React + Tauri（桌面應用）

- **定位**：真正的「跨平台桌面應用」
- **跨平台方式**：打包成 `.exe` / `.dmg` / `.deb` 安裝檔
- **存取系統資源**：完整支援（檔案系統、通知、系統托盤等）
- **效能**：極佳（Rust 後端，比 Electron 輕很多，約 3–10 MB）
- **缺點**：需要安裝 Rust 環境，初次設定複雜
- **適合**：桌面工具、需要存取本機資源的應用

## 二、Deno 是什麼？為何需要它？

### Deno vs Node.js 核心差異

| 項目 | Node.js | Deno |
|------|---------|------|
| 安全性 | 預設可存取所有資源 | 預設全部拒絕，需明確授權 |
| TypeScript | 需要額外設定 | 原生支援，直接跑 `.ts` |
| 套件管理 | 需要 `npm install` + `node_modules` | 直接 `import` URL，或用 `deno add` |
| 標準函式庫 | 依賴第三方套件 | 內建完整標準函式庫 |
| 設定檔 | 需要 `tsconfig.json`、`package.json`... | 一個 `deno.json` 搞定 |

### Deno 在架構中的角色

Deno 主要負責「後端」的部分，讓你不需要再學 Node.js + Express 那套：

```
加入 Deno 後的架構：
前端（Vite + React）→ 打包後的靜態檔案
後端（Deno）        → 提供 API、處理資料庫、驗證身份
兩者部署到同一台伺服器
```

### 重要判斷：你需要 Deno 嗎？

**如果沒有後端需求**（純前端）→ 純 Vite + React 就夠了，完全不需要 Deno

**如果有以下需求**，才考慮加入 Deno：
- 需要登入 / 會員系統
- 需要資料庫（存使用者資料）
- 需要 API Server

## 三、三大技術堆疊完整比較

### Deno + Vite + React

**架構組成：**
- 前端：React + TSX、Vite HMR
- 後端：Deno runtime、安全沙箱、原生 TS 支援
- 部署：任何 Linux 伺服器、Deno Deploy、Docker

**優點：**
- TypeScript 原生支援，不需要設定（Deno 直接執行 `.ts` 檔案）
- 安全性更高（預設拒絕存取檔案、網路、環境變數）
- 啟動速度快、設定簡潔

**缺點：**
- npm 生態系相容性較差（雖然 Deno 2.0 後大幅改善）
- 社群與教學資源較少，遇到問題較難找答案

**最適合**：想嘗鮮新技術、對安全性有要求、不需要依賴太多複雜 npm 套件

---

### Node.js + Vite + React

**架構組成：**
- 前端：React + TSX、Vite HMR、npm 生態系
- 後端：Node.js + Express、資料庫連線、REST API
- 部署：任何 Linux 伺服器、Railway / Render、Docker

**優點：**
- 生態系最成熟、教學資源最多
- 對前端工程師最友善，只需再學後端的 Express 部分
- 幾乎所有 npm 套件都完整支援

**缺點：**
- 前後端是兩個獨立專案，需要分開管理、部署
- 跨域（CORS）問題要自己處理
- 設定檔較多（tsconfig.json、.eslintrc 前後端各一份）

**最適合**：已熟悉 Vite + React、需要後端 API、想用最多教學資源輔助學習

> 對已有 Vite + React 經驗的前端工程師，這是學習成本最低的選項

---

### Next.js

**架構組成：**
- 前端：React + TSX、SSR / SSG / CSR、App Router
- 後端（同一個專案）：Route Handlers / API、Server Components、直接存取 DB
- 部署：Vercel（最佳化）、任何 Node 伺服器、Docker

**優點：**
- 前後端在同一個專案，不需要管 CORS、不需要分開部署
- SEO 與效能最強（SSR / SSG 首頁載入快、搜尋引擎可爬到內容）
- TypeScript 內建支援

**缺點：**
- 概念最多，學習曲線最陡（Server Components、Client Components、SSR、SSG、ISR…）
- 跟 Vercel 綁定較深，部署到其他伺服器需要額外設定

**最適合**：需要 SEO 的官網、電商、內容型網站，或想一個框架搞定全端開發

---

### 綜合比較表

| 項目 | Deno + Vite | Node.js + Vite | Next.js |
|------|------------|----------------|---------|
| 執行環境 | Deno runtime | Node.js | Node.js |
| 前後端關係 | 分離（兩個 port） | 分離（兩個 port） | 整合（同一個專案） |
| SEO 支援 | 需額外設定 | 需額外設定 | 原生支援 |
| TypeScript | 零設定原生支援 | 需要設定 | 內建 |
| 學習難度 | ⭐⭐ 中 | ⭐ 低 | ⭐⭐⭐ 高 |
| npm 相容性 | 部分相容 | 完整相容 | 完整相容 |
| 教學資源 | 較少 | 最多 | 很多 |

## 四、如何選擇適合你的技術組合

### 決策流程

```
你的應用有後端需求嗎？（登入、資料庫、API）
│
├── 沒有 → 繼續用你熟悉的 Vite + React 就好
│
└── 有 → 需要 SEO 嗎？（官網、部落格、電商）
          │
          ├── 需要 → Next.js
          │          前後端一體、SEO 最強
          │
          └── 不需要 → 想嘗試新技術嗎？
                        │
                        ├── 是 → Deno + Vite + React
                        │
                        └── 否 → Node.js + Vite + React（推薦）
                                  學習成本最低、資源最多
```

### 按情境推薦

| 你的情況 | 推薦方案 | 理由 |
|---------|---------|------|
| 純前端，不需要後端 | Vite + React（現有技能） | 完全不需要改變 |
| 需要後端 API、重視教學資源 | Node.js + Vite + React | 學習成本最低 |
| 需要 SEO、官網、內容型網站 | Next.js | 長期最值得投資 |
| 想嘗試新技術、重視安全性 | Deno + Vite + React | 未來性佳但資源少 |
| 做桌面工具 App | Deno / Node + Vite + React + Tauri | 唯一能打包成安裝檔的選項 |
